# Ollama 任务分析提示词 v9.0 - 优化方案

## 核心改进思路

将「规则描述」转换为「语义特征描述」，让 Ollama 理解：
- L0: 询问时间信息（无需操作）
- L1: 单步只读查询（直接执行）
- L2: 思考分析规划（产物是文档）
- L3: 执行操作意图（产物是方案，等确认）
- L4: 确认执行方案（实际执行）

---

## 语义特征识别

### L0 - 时间询问

**语义特征**：
- 询问时间、日期、星期等时间信息
- 不需要任何工具，直接回答
- 单句完整问题，无上下文依赖

**识别关键词**：几点、时间、日期、星期、今天、明天、昨天、现在

**语义模式**：「现在几点」→ 无需操作 → 直接回答

---

### L1 - 只读查询

**语义特征**：
- 单步只读操作
- 查看、读取、搜索、查找
- 不修改任何文件或状态
- 不涉及复杂逻辑推理

**识别关键词**：查看、读取、显示、列出、搜索、查找、获取

**语义模式**：「查看X」→ 单步只读 → 直接执行

**区分点**：不涉及「分析」「思考」「规划」等深度认知

---

### L2 - 思考分析

**语义特征**：
- 需要深度思考、推理、分析
- 制定方案、规划设计、评估比较
- 产物是「分析报告」或「方案文档」
- 不涉及实际代码/配置的创建或修改

**识别关键词**：分析、比较、评估、设计、优化、思考、制定方案、规划

**语义模式**：「分析X」→ 深度思考 → 输出分析结论

**区分点**：产物是「分析报告」「方案文档」，不是「代码文件」「配置文件」

---

### L3 - 执行意图

**语义特征**：
- 用户想要执行某个操作（创建/修改/删除/部署等）
- AI 的任务是：先生成执行方案，等待用户确认
- L3 阶段的产物是「方案文档」，不是「实际结果」
- **write_operation=false**（方案不是实际写入）

**识别关键词**：创建、修改、删除、部署、安装、卸载、写入、配置

**语义模式**：「创建X」→ 先生成方案 → 等待确认 → (用户确认后进入L4)

**区分点**：用户意图是「执行」，但 AI 必须先出方案

**关键区分**：
- 「制定优化方案」→ L2（产物是方案文档）
- 「优化代码」→ L3（涉及执行优化）
- 「生成分析报告」→ L2（产物是报告文档）
- 「修复bug」→ L3（涉及执行修复）
- 「创建一个优化方案」→ L2（产物是方案文档，不是创建文件）

---

### L4 - 确认执行

**语义特征**：
- 用户明确表示同意、确认
- 上下文存在已生成的执行方案
- 方案已被用户确认，进入执行阶段
- **write_operation=true**（实际执行）

**识别关键词**：同意、确认、确认执行、开始吧、好的、可以、没问题、approve、confirm、yes、ok

**语义模式**：「同意」+ 存在方案 → 执行方案

**前置条件**：存在 execution_plan.md

---

## L2 vs L3 判断规则

**核心区分**：看「产物」是什么

| 维度 | L2 | L3 |
|------|----|----|
| 用户意图 | 思考、分析、规划 | 执行操作 |
| AI 产物 | 分析报告、方案文档 | 执行方案（execution_plan.md） |
| 是否写入 | 否 | 否（方案不是写入） |
| 后续动作 | 结束 | 等用户确认 → L4 |

**判断规则**：
```
用户说「制定优化方案」→ 产物是「方案文档」→ L2
用户说「优化代码」→ 涉及「执行优化」→ L3（出优化方案）

用户说「生成分析报告」→ 产物是「报告文档」→ L2
用户说「修复bug」→ 涉及「执行修复」→ L3（出修复方案）
```

## 常见误判纠正

| 用户说 | 错误判断 | 正确判断 | 语义分析 |
|--------|---------|---------|---------|
| 「制定优化方案」 | L3 | L2 | 产物是方案文档，不是执行 |
| 「生成分析报告」 | L3 | L2 | 产物是报告文档，不是执行 |
| 「创建一个优化方案」 | L3 | L2 | 产物是方案文档，不是创建文件 |
| 「分析并修复」 | L2 | L3 | 包含「修复」执行意图 |
| 「查看配置」 | L3 | L1 | 单步只读查询 |
| 「修改配置」 | L2 | L3 | 涉及修改操作意图 |

## write_operation 语义判断

### true 的情况：
- L4 阶段：实际执行方案，会写入代码/配置
- 用户明确说「直接执行，跳过方案」

### false 的情况：
- L0/L1：只读查询
- L2：输出分析文档（不是写入）
- L3：生成执行方案（方案不是实际写入）

**关键理解**：
- L3 的「方案」不是「写入」
- 只有 L4 的「执行」才是「写入」

---

## 语义模式识别

### 模式 1: 询问信息 → L0/L1
- 「现在几点」→ 无需操作 → L0
- 「查看文件」→ 单步只读 → L1

### 模式 2: 思考分析 → L2
- 「分析X」→ 深度思考 → 输出分析结论 → L2
- 「制定方案」→ 规划设计 → 输出方案文档 → L2

### 模式 3: 执行操作 → L3
- 「创建X」→ 生成执行方案 → 等待确认 → L3
- 「修改X」→ 生成执行方案 → 等待确认 → L3

### 模式 4: 确认执行 → L4
- 「同意」+ 存在方案 → 执行方案 → L4

---

## L3 → L4 转换流程

```
# L3 阶段
用户：「创建配置文件」
输出：{"task_level": "L3", "write_operation": false, "phase_action": "continue"}
AI 调用 openclaw-behavior-plan → 生成 execution_plan.md → 输出方案

# 用户确认后
用户：「同意」
输出：{"task_level": "L4", "write_operation": true, "phase_action": "transition"}
AI 调用 planning-with-files + agent-pool → 执行方案
```

## 关键改进总结

### 1. L3 定义修正
**之前**：L3 = 创建/修改/删除（暗示直接执行）
**现在**：L3 = 执行意图（先生成方案，等确认）

### 2. write_operation 语义修正
**之前**：L3 的 write_operation=true
**现在**：L3 的 write_operation=false（方案不是实际写入）

### 3. L2 vs L3 边界明确
**核心判断**：看「产物」是什么
- 产物是「文档」→ L2
- 产物是「方案+等确认」→ L3
