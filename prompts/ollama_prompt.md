# Ollama 任务分析提示词 v8.0

分析用户消息，返回 JSON 格式的分析结果。

{phase_context}

用户消息：
{user_message}

---

## 输出格式

返回 JSON（仅返回 JSON，不要其他内容）：

```json
{
    "optimized_prompt": "整合后的文本",
    "task_level": "L0/L1/L2/L3/L4",
    "phase_action": "continue/transition/reset",
    "write_operation": true/false,
    "code_guidance": true/false,
    "agent_pool": true/false,
    "skill_usage": true/false,
    "self_improving": true/false
}
```

## optimized_prompt 规则

**核心目标**：优化用户输入，使 AI 能清楚理解、高效执行。

### 判断流程（完整）

```
第一步：检测结构化内容（必须保留）
├── 表格（|---|）→ 保持原样（结构化数据，AI 需要完整信息）
├── 代码块（```）→ 保持原样（代码语法不可破坏）
├── 列表（-/1.）→ 保持原样（层级关系不可打乱）
├── 标题（#）→ 保持原样（文档结构不可改变）
└── JSON/YAML → 保持原样（配置数据不可修改）

第二步：消息类型处理（语义优化）
├── 纯结构化 → 直接返回原文（无冗余，无需优化）
├── 口语+结构 → 提取意图 + 保持结构（去除口语，保留关键信息）
└── 纯口语化 → 结构化改写（提取核心指令，明确动词和对象）
```

### 语义优化规则（针对纯口语化输入）

**优化目标**：让 AI 一眼看清「做什么」「对什么」「怎么做」

| 原始口语化输入 | 优化后输出 | 优化说明 |
|---------------|-----------|---------|
| 「帮我看看这个文件」 | 「读取文件内容」 | 明确动词 + 对象 |
| 「能不能把这里改一下」 | 「修改指定位置」 | 明确操作意图 |
| 「我觉得这里可以优化一下」 | 「优化指定内容」 | 提取核心指令 |
| 「就是那个...你懂的」 | 「（保持原文，无法推断）」 | 信息不足则不改 |
| 「分析一下这个项目的结构，然后告诉我有什么问题」 | 「分析项目结构并识别问题」 | 合并多句为单指令 |
| 「如果是 A 就做 X，如果是 B 就做 Y，然后都再检查一下」 | 「条件分支：A→X，B→Y，最后验证结果」 | 结构化逻辑表达 |

### 优化原则

1. **保留关键信息**：动词、对象、条件、目标
2. **去除冗余**：寒暄、犹豫词、重复表达
3. **结构化表达**：复杂逻辑用表格/列表/条件分支
4. **信息不足不改**：无法推断意图时，保持原文
5. **显式锚定**：若输入包含代词（这/那/它/刚才），且无法从**当前对话轮次直接唯一确定**，需将代词替换为 `[需根据上文补全]` 或保留原文并在括号内标注推测对象。
6. **偏好量化**：将“随便/好看/高级感”等抽象审美词，替换为对应的**设计参数描述**（如：饱和度低、留白多、无衬线字体）。



## 任务等级语义定义

| 等级 | 用户意图 | 特征 | 示例 |
|------|---------|------|------|
| L0 | 问时间/日期 | 不需要任何操作，直接回答 | 现在几点、今天是星期几 |
| L1 | 查看/读取/搜索 | 单步只读，无副作用 | 查看文件、搜索内容、读取配置 |
| L2 | 分析/思考/规划 | 需要深度思考，无写入操作 | 分析项目、比较方案、制定方案 |
| L3 | 创建/修改/删除 | 需要执行操作，有副作用 | 创建文件、修改配置、部署服务 |
| L4 | 确认执行 | 用户确认后执行已有方案 | 同意、确认、执行、开始吧 |

## 核心判断：问自己三个问题

### 问题 1：用户想做什么？

```
只是问信息 → L0/L1
需要思考分析 → L2
需要执行操作 → L3
确认已有方案 → L4
```

### 问题 2：是否涉及写入？

```
修改代码/配置/文件 → write_operation=true
只写方案文档/报告 → write_operation=false
只读操作 → write_operation=false
```

### 问题 3：任务复杂度？

```
单步查询 → L1
需要思考分析 → L2
需要执行多个操作 → L3
用户确认执行 → L4
```

---

## L2 vs L3 关键区别

| 维度 | L2 思考任务 | L3 执行任务 |
|------|------------|------------|
| 目的 | 思考、分析、设计、评估 | 创建、修改、删除、部署 |
| 产物 | 方案文档、分析报告 | 代码文件、配置文件 |
| 关键词 | 制定方案、分析、评估、设计 | 创建、修改、删除、部署 |
| 副作用 | 无 | 有 |

**关键判断**：
- 「制定方案」→ L2（方案是规划文档，不是代码）
- 「生成报告」→ L2（报告是分析文档）
- 「创建文件」→ L3（实际创建文件）
- 「修改配置」→ L3（实际修改配置）

---

## L4 判断规则

**前置条件**：
1. 用户消息包含确认词：「同意」「确认」「执行」「开始吧」「好的」「可以」「没问题」「approve」「confirm」「yes」「ok」
2. 上下文显示存在 execution_plan.md 或当前阶段为 L3 completed

**判断**：
- 确认词 + 存在方案 → L4
- 确认词但无方案 → L3

## 常见误判纠正

| 用户说 | 错误判断 | 正确判断 | 原因 |
|--------|---------|---------|------|
| 「制定方案」 | L3 | L2 | 方案是规划文档，不是代码执行 |
| 「生成报告」 | L3 | L2 | 报告是分析文档，无副作用 |
| 「分析并优化」 | L3 | L2 | 分析阶段，未执行优化 |
| 「设计方案」 | L3 | L2 | 设计是规划，不是实现 |
| 「帮我思考一下」 | L3 | L2 | 思考是分析，不是执行 |

---

## 阶段动作判断

| phase_action | 触发条件 |
|--------------|----------|
| continue | 正常执行，保持当前阶段 |
| transition | 阶段转换（L2→L3 或 L3→L4） |
| reset | 用户说「新任务」「取消」「重置」「算了」 |


## 检测规则

你需要判断用户消息属于哪种场景，然后返回 true 或 false。

### write_operation

**问自己**：用户想要修改什么吗？

用户说类似这些话，返回 true：
- 创建、修改、删除文件
- 部署、安装、卸载
- 修改配置、更改设置

用户说类似这些话，返回 false：
- 查看、读取、搜索
- 分析、思考、规划
- 制定方案、生成报告（方案和报告是分析产物，不是修改文件）


### code_guidance

**问自己**：用户在谈论代码吗？

用户提到这些词，返回 true：
- 代码、编程、调试
- 函数、类、模块
- 算法、逻辑实现

用户在说这些事，返回 false：
- 分析项目结构（结构分析不是代码本身）
- 查看配置文件（配置不是代码）
- 制定方案、规划（规划不是编程）
- 数据分析、文档分析

例子：
- 「修改这个函数」→ true（涉及函数）
- 「调试代码报错」→ true（涉及调试）
- 「分析项目结构」→ false（结构分析）
- 「查看配置文件」→ false（配置不是代码）
- 「制定方案」→ false（规划不是编程）

---

### agent_pool

**问自己**：用户要同时做很多事吗？

用户说这些话，返回 true：
- 同时分析多个项目
- 并行处理多个任务
- 批量处理多个文件

其他情况，返回 false：
- 单一分析任务
- 查看单个文件
- 制定一个方案

例子：
- 「同时分析3个项目」→ true（多个项目）
- 「并行下载这些文件」→ true（批量操作）
- 「分析项目结构」→ false（单一任务）
- 「查看文件」→ false（简单任务）
- 「制定方案」→ false（单线程思考）

默认情况：大多数任务都是单一的，返回 false

---

### skill_usage

**问自己**：用户的问题可以使用技能快速完成吗？

用户直接说技能名，返回 true：
- 用 deep-thinking 分析
- 调用 workflow-manager
- 使用 agent-pool 执行

默认为true

---

### self_improving

**问自己**：用户想让 AI 学习或改进吗？

用户说类似这些话，返回 true：
- 记住这个偏好
- 纠正你的错误
- 下次不要这样
- 优化这个流程
- 更新记忆

其他情况，返回 false：
- 常规的分析、查询任务
- 文件操作
- 方案制定
