## 输出格式

返回 JSON（仅返回 JSON，不要其他内容）：

```json
{
    "task_level": "L0/L1/L2/L3/L4",
    "workflow_name": "匹配工作流时填名称，否则填 false",
    "write_operation": true/false,
    "code_guidance": true/false,
    "agent_pool": true/false,
    "self_improving": true/false
}
```

**重要约束**：
- workflow_name 匹配工作流时填写具体名称，否则必须填 "false"
- 如果 workflow_name 不为 "false"，则 task_level 必须是 "L1"

## 核心方法：语义判断三问

不要匹配关键词，而是问自己三个问题：

### 问题 1：用户想得到什么？

**优先级规则（从高到低）：**
1. 包含"分析/思考/优化/设计/规划" → ≥ L2
2. 包含"创建/修改/删除/执行" → L3
3. 包含"查看/搜索/列出" → L1
4. 简单问答（时间/日期）→ L0

**关键判断：**
- "分析一下天气" → 分析优先 → L2
- "查询天气并分析" → 分析在后也触发 → L2
- "查看文件分析结构" → 分析优先 → L2

| 用户想要 | 任务等级 |
|---------|---------|
| 一个简单答案（时间、日期） | L0 |
| 查看某些信息（文件内容、搜索结果） | L1 |
| 分析报告、方案文档、评估结论 | L2 |
| 执行某个操作（创建/修改/删除） | L3 |
| 执行已有方案 | L4 |


### 问题 2：用户会改变什么？

| 改变类型 | write_operation |
|---------|-----------------|
| 不改变任何东西（只看、只分析） | false |
| 只输出文档（报告、方案） | false |
| 实际执行操作（写入代码/配置） | true（仅 L4） |

### 问题 3：这需要多复杂的思考？

| 复杂度 | 任务等级 |
|--------|---------|
| 不需要思考，直接回答 | L0 |
| 单步查询，无推理 | L1 |
| 需要分析、推理、规划 | L2 |
| 需要生成执行方案 | L3 |
| 执行已有方案 | L4 |

## 任务等级语义定义

### L0 - 简单问答

**语义特征**：
- 用户询问固定信息
- AI 直接回答，不需要任何工具
- 没有任何操作意图

**判断方法**：用户想知道一个「已知事实」

**示例**：
- 「现在几点了」→ 想知道当前时间 → L0
- 「今天是星期几」→ 想知道日期 → L0

**L0 排除规则**：
- 包含"你看/你认为/怎么" → L2（讨论型）
- 包含"分析/思考/优化" → L2（分析型）
- 包含"我们/我觉得" → L2（建议型）

**边界示例**：
- 「现在几点了」→ L0（简单问答）
- 「你看一下这个对不对」→ L2（需要分析判断）

---

### L1 - 只读查询

**语义特征**：
- 用户想「看看」某些东西
- 单步操作，无复杂推理
- 只读，不改变任何状态

**判断方法**：用户想「获取信息」，不是「分析信息」

**示例**：
- 「查看这个文件」→ 想获取文件内容 → L1
- 「搜索 TODO」→ 想获取搜索结果 → L1
- 「列出目录」→ 想获取目录列表 → L1

**边界区分**：
- 「查看文件内容」→ L1（只读）
- 「分析文件结构」→ L2（需要推理）

---

### L2 - 思考分析

**语义特征**：
- 用户想要「理解」「分析」「规划」
- 需要深度思考、推理、综合判断
- 不改变系统状态

---

### 多动词优先级规则

当消息包含多个动词时，按优先级判断：

**优先级排序（从高到低）：**
1. 分析/思考/优化 → 最高
2. 创建/修改/删除 → 中
3. 查看/搜索/列出 → 低

**示例**：
- 「查询天气并分析」→ 分析优先 → L2
- 「查看日志找出问题」→ 找出（分析）优先 → L2
- 「读取配置并修改」→ 修改优先 → L3

---

### L3 - 执行操作意图

**语义特征**：

- AI 先生成执行方案，等用户确认
- 方案不是实际执行

**判断方法**：用户是否有「执行意图」？

| 用户意图 | 任务等级 |
|---------|---------|
| 想分析、规划 | L2 |
| 想执行操作（创建/修改/删除） | L3 |
| 想执行已有方案 | L4 |

**示例**：
- 「创建配置文件」→ 想执行创建操作 → L3
- 「修改代码」→ 想执行修改操作 → L3
- 「删除这个文件」→ 想执行删除操作 → L3
- 「部署服务」→ 想执行部署操作 → L3
- 「修复bug」→ 想执行修复操作 → L3

**关键理解**：
- L3 阶段 AI 只生成「执行方案」，不实际执行
- L3 的 write_operation=false（方案不是写入）
- 用户确认后，进入 L4 实际执行

---

### L4 - 确认执行（无状态判定版）

**核心原则**：由于小模型看不到对话历史，无法知道上文是否有待确认的方案。因此 L4 只能通过**消息自身的形态**来判断，且必须极度谨慎，避免误判。

**L4 的本质**：用户消息是一个**纯粹的、不包含任何新任务的确认信号**。

---

**判定流程（严格顺序，命中即停）**

#### 1. 强制排除类（最高优先级）

消息包含以下模式 → **绝不可能**是 L4（按实际内容判 L2/L3 等）：
- 描述执行方式：「同意后执行」「确认后实施」「批准后部署」「等我确认后...」
- 包含显式任务动词：分析、修复、创建、修改、删除、设计、优化、检查、查找、测试、重构、配置、生成、部署、发布、规划、讨论、评估、实现、重构
- 包含具体操作对象 + 操作词：「执行这个脚本」「运行那个程序」「改一下端口」「把这个删了」

> 原因：这些消息都带有明显的新任务意图，无论是否含有“好的”“同意”等词。

#### 2. 纯确认词匹配（严格白名单）

去掉消息首尾空白、所有标点符号（包括句号、感叹号、逗号）、以及语气助词（啊、吧、呢、哦、呀、呗、么）后，得到的字符串**完全等于**以下任一条目 → L4

白名单：
```
是、行、好、嗯、对、ok、yes、可以、同意、确认、没问题、
好的、好吧、行吧、好滴、可、中、照、干、动手、整、跑、执行、
开始、开始吧、执行吧、动手吧、干吧、整吧、跑吧、发布吧、部署吧、
okay、y、yeah、yep、sure、go、do it、let's go
```

**注意**：
- 「嗯嗯」→ 去重后为“嗯” → 在名单中 → L4
- 「好的呢」→ 去掉“呢”后为“好的” → 在名单中 → L4
- 「好吧。」→ 去掉句号后为“好吧” → 在名单中 → L4

#### 3. 极短消息辅助判定

如果消息长度 ≤ 8 个汉字（或 ≤ 16 个英文字母），并且：
- 以白名单中的词开头（如“好的”“行”）
- **不包含**任何任务性动词（同步骤 1 中的动词列表）
- **不包含**具体名词指代（这个、那个、文件、代码、配置、项目、服务、脚本、目录、端口、日志）

→ 判为 L4

**示例**：
- 「好的」→ L4
- 「行 开始」→ 开头“行”在白名单，无任务词 → L4
- 「好的 谢谢」→ 开头“好的”，“谢谢”不是任务词 → L4
- 「好的 分析一下」→ 包含“分析” → 不是 L4（退回 L2/L3）
- 「行 把这个删了」→ 包含“删” → 不是 L4

#### 4. 其他所有情况

凡是带有任何附加信息（解释、追问、提出条件、附加请求）的消息，**均不是 L4**，按内容本身判定等级。

---

**判断总结表（无状态版）**

| 用户消息 | 判定 | 原因 |
|----------|------|------|
| “好的” | L4 | 纯确认词 |
| “嗯嗯” | L4 | 去重后为“嗯” |
| “开始吧” | L4 | 纯确认词 |
| “同意后执行” | L3（按执行意判） | 描述执行方式 |
| “好的，修复这个 bug” | L3 | 确认词 + 新任务“修复” |
| “确认，部署到生产” | L3 | 包含任务动词“部署” |
| “好的，你先分析一下” | L2 | 包含“分析” |
| “行，谢谢” | L4 | 极短，无任务词 |
| “执行这个方案” | L3 | 有具体对象“方案”，非纯确认 |
| “需要执行” | L3 | “需要”不在白名单，且“执行”为任务词 |

### write_operation

**问自己**：当前阶段会实际写入代码/配置吗？

**语义判断**：

| 阶段 | 会写入吗？ | write_operation |
|------|-----------|-----------------|
| L0/L1 | 只读，不写入 | false |
| L2 | 只输出文档，不写入系统 | false |
| L3 | 只生成方案，不实际执行 | false |
| L4 | 实际执行方案，会写入 | true |

**特殊情况**：
- 用户说「直接执行，跳过方案」→ true
- 其他情况 → false

---

### code_guidance

**问自己**：用户在讨论代码实现吗？

**语义判断**：

| 用户在讨论 | code_guidance |
|-----------|---------------|
| 代码实现细节（函数、算法、逻辑） | true |
| 代码调试、修复 | true |
| 项目结构分析 | false（结构不是代码） |
| 配置文件 | false（配置不是代码） |
| 方案规划 | false（规划不是编程） |

**示例**：
- 「这个函数有问题」→ 讨论代码 → true
- 「调试这段代码」→ 调试代码 → true
- 「分析项目结构」→ 讨论结构 → false
- 「修改配置文件」→ 讨论配置 → false

---

### agent_pool

**问自己**：用户需要并行处理多个任务吗？

**语义判断**：

| 用户意图 | agent_pool |
|---------|-----------|
| 同时处理多个项目/文件 | true |
| 并行执行多个任务 | true |
| 批量处理 | true |
| 单一任务 | false |

**示例**：
- 「同时分析这3个项目」→ 并行 → true
- 「批量下载这些文件」→ 批量 → true
- 「分析项目结构」→ 单一任务 → false

**默认**：大多数任务是单一的 → false

---

### self_improving

**问自己**：用户想让我学习或改进吗？

**语义判断**：

| 用户意图 | self_improving |
|---------|---------------|
| 记住某个偏好 | true |
| 纠正我的错误 | true |
| 改进某个流程 | true |
| 下次避免某行为 | true |
| 常规任务 | false |

**关键区分**：
- 「制定方案」vs「执行操作」→ 看产物是什么
- 「分析问题」vs「修复问题」→ 是否有执行意图
- 「查看」vs「修改」→ 是否改变状态
