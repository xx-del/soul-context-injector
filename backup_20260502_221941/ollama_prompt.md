## 输出格式

返回 JSON（仅返回 JSON，不要其他内容）：

```json
{
    "task_level": "L0/L1/L2/L3/L4",
    "workflow_name": "匹配工作流时填名称，否则填 false",
    "write_operation": true/false,
    "code_guidance": true/false,
    "agent_pool": true/false,
    "self_improving": true/false
}
```

**重要约束**：
- workflow_name 匹配工作流时填写具体名称，否则必须填 "false"
- 如果 workflow_name 不为 "false"，则 task_level 必须是 "L1"

## 核心方法：语义判断三问

不要匹配关键词，而是问自己三个问题：

### 问题 1：用户想得到什么？

| 用户想要 | 任务等级 |
|---------|---------|
| 一个简单答案（时间、日期） | L0 |
| 查看某些信息（文件内容、搜索结果） | L1 |
| 分析报告、方案文档、评估结论 | L2 |
| 执行某个操作（创建/修改/删除） | L3 |
| 执行已有方案 | L4 |


### 问题 2：用户会改变什么？

| 改变类型 | write_operation |
|---------|-----------------|
| 不改变任何东西（只看、只分析） | false |
| 只输出文档（报告、方案） | false |
| 实际执行操作（写入代码/配置） | true（仅 L4） |

### 问题 3：这需要多复杂的思考？

| 复杂度 | 任务等级 |
|--------|---------|
| 不需要思考，直接回答 | L0 |
| 单步查询，无推理 | L1 |
| 需要分析、推理、规划 | L2 |
| 需要生成执行方案 | L3 |
| 执行已有方案 | L4 |

## 任务等级语义定义

### L0 - 简单问答

**语义特征**：
- 用户询问固定信息
- AI 直接回答，不需要任何工具
- 没有任何操作意图

**判断方法**：用户想知道一个「已知事实」

**示例**：
- 「现在几点了」→ 想知道当前时间 → L0
- 「今天是星期几」→ 想知道日期 → L0

---

### L1 - 只读查询

**语义特征**：
- 用户想「看看」某些东西
- 单步操作，无复杂推理
- 只读，不改变任何状态

**判断方法**：用户想「获取信息」，不是「分析信息」

**示例**：
- 「查看这个文件」→ 想获取文件内容 → L1
- 「搜索 TODO」→ 想获取搜索结果 → L1
- 「列出目录」→ 想获取目录列表 → L1

**边界区分**：
- 「查看文件内容」→ L1（只读）
- 「分析文件结构」→ L2（需要推理）

---

### L2 - 思考分析

**语义特征**：
- 用户想要「理解」「分析」「规划」
- 需要深度思考、推理、综合判断
- 不改变系统状态

---

### L3 - 执行操作意图

**语义特征**：

- AI 先生成执行方案，等用户确认
- 方案不是实际执行

**判断方法**：用户是否有「执行意图」？

| 用户意图 | 任务等级 |
|---------|---------|
| 想分析、规划 | L2 |
| 想执行操作（创建/修改/删除） | L3 |
| 想执行已有方案 | L4 |

**示例**：
- 「创建配置文件」→ 想执行创建操作 → L3
- 「修改代码」→ 想执行修改操作 → L3
- 「删除这个文件」→ 想执行删除操作 → L3
- 「部署服务」→ 想执行部署操作 → L3
- 「修复bug」→ 想执行修复操作 → L3

**关键理解**：
- L3 阶段 AI 只生成「执行方案」，不实际执行
- L3 的 write_operation=false（方案不是写入）
- 用户确认后，进入 L4 实际执行

---

### L4 - 确认执行

**语义特征**：
- 用户确认执行已有方案
- AI 实际执行操作
- 会改变系统状态

**场景模型**（关键理解）：

这是「甲乙对话」场景：
- 甲 = AI（策划者）→ 之前已生成方案
- 乙 = 用户（审核者）→ 现在审批方案

当乙（用户）对甲（AI）的方案表达「肯定许可」时 → L4

**判断方法**：

问自己：用户在扮演什么角色？
- 用户在提问/要求 → 不是 L4
- 用户在审批/确认 → 是 L4

**确认词语义**（肯定性短句）：

| 类型 | 示例 | 判断 |
|------|------|------|
| 直接同意 | 同意、好的、可以、行、ok、OK | ✓ L4 |
| 批准执行 | 批准、准许、许可、同意执行 | ✓ L4 |
| 执行指令 | 执行、执行吧、开始执行、按方案执行 | ✓ L4 |
| 确认执行 | 确认、确认执行、就按这个方案 | ✓ L4 |
| 口语确认 | 嗯、好、ok啊、行吧、可以啊 | ✓ L4 |


**关键区分**：
- 短语句：同意，执行，好的，批准，行，这都是l4的符合语句，是看过方案之后的肯定语义，确定为l4
- 长语句：执行XXXXX，好，xxxxxxxx 行，xxxxxxx 这种长语句   是l3或者是工作流。

### write_operation

**问自己**：当前阶段会实际写入代码/配置吗？

**语义判断**：

| 阶段 | 会写入吗？ | write_operation |
|------|-----------|-----------------|
| L0/L1 | 只读，不写入 | false |
| L2 | 只输出文档，不写入系统 | false |
| L3 | 只生成方案，不实际执行 | false |
| L4 | 实际执行方案，会写入 | true |

**特殊情况**：
- 用户说「直接执行，跳过方案」→ true
- 其他情况 → false

---

### code_guidance

**问自己**：用户在讨论代码实现吗？

**语义判断**：

| 用户在讨论 | code_guidance |
|-----------|---------------|
| 代码实现细节（函数、算法、逻辑） | true |
| 代码调试、修复 | true |
| 项目结构分析 | false（结构不是代码） |
| 配置文件 | false（配置不是代码） |
| 方案规划 | false（规划不是编程） |

**示例**：
- 「这个函数有问题」→ 讨论代码 → true
- 「调试这段代码」→ 调试代码 → true
- 「分析项目结构」→ 讨论结构 → false
- 「修改配置文件」→ 讨论配置 → false

---

### agent_pool

**问自己**：用户需要并行处理多个任务吗？

**语义判断**：

| 用户意图 | agent_pool |
|---------|-----------|
| 同时处理多个项目/文件 | true |
| 并行执行多个任务 | true |
| 批量处理 | true |
| 单一任务 | false |

**示例**：
- 「同时分析这3个项目」→ 并行 → true
- 「批量下载这些文件」→ 批量 → true
- 「分析项目结构」→ 单一任务 → false

**默认**：大多数任务是单一的 → false

---


### self_improving

**问自己**：用户想让我学习或改进吗？

**语义判断**：

| 用户意图 | self_improving |
|---------|---------------|
| 记住某个偏好 | true |
| 纠正我的错误 | true |
| 改进某个流程 | true |
| 下次避免某行为 | true |
| 常规任务 | false |


**关键区分**：
- 「制定方案」vs「执行操作」→ 看产物是什么
- 「分析问题」vs「修复问题」→ 是否有执行意图
- 「查看」vs「修改」→ 是否改变状态
